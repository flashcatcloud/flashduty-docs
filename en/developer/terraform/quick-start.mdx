## Prerequisites

Before you begin, make sure you have the following ready:

- **Terraform >= 1.0** — If not yet installed, refer to the [official Terraform installation guide](https://developer.hashicorp.com/terraform/downloads).
- **Flashduty APP Key** — Obtain it from "Account Settings -> APP Key" in the Flashduty console. This key is used by the Provider to call the Flashduty API.

<Warning>The APP Key has account-level management permissions. Keep it safe and do not hardcode it in public code repositories. It is recommended to pass it via environment variables.</Warning>

## Configure the Provider

<Steps>
  <Step title="Create a project directory">
    Create a new directory to store Terraform configuration files:

    ```bash
    mkdir flashduty-terraform && cd flashduty-terraform
    ```
  </Step>
  <Step title="Create a configuration file">
    Create a `main.tf` file to declare the Provider source and authentication configuration:

    <CodeGroup>
    ```hcl main.tf (environment variable method, recommended)
    terraform {
      required_providers {
        flashduty = {
          source  = "flashcatcloud/flashduty"
          version = "~> 0.1"
        }
      }
    }

    provider "flashduty" {
      # APP Key is passed via the FLASHDUTY_APP_KEY environment variable
    }
    ```

    ```hcl main.tf (direct configuration method)
    terraform {
      required_providers {
        flashduty = {
          source  = "flashcatcloud/flashduty"
          version = "~> 0.1"
        }
      }
    }

    provider "flashduty" {
      app_key = "your-app-key"
    }
    ```
    </CodeGroup>

    <Tip>It is recommended to pass the APP Key via environment variables to avoid exposing sensitive information. Set it with: `export FLASHDUTY_APP_KEY="your-app-key"`</Tip>
  </Step>
  <Step title="Initialize the Provider">
    Run `terraform init` to download and install the Flashduty Provider:

    ```bash
    terraform init
    ```

    On success, you will see output similar to the following:

    ```
    Initializing provider plugins...
    - Finding flashcatcloud/flashduty versions matching "~> 0.1"...
    - Installing flashcatcloud/flashduty v0.1.x...
    - Installed flashcatcloud/flashduty v0.1.x

    Terraform has been successfully initialized!
    ```
  </Step>
</Steps>

### Provider parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `app_key` | String | No | Flashduty APP Key, can also be set via the `FLASHDUTY_APP_KEY` environment variable |
| `base_url` | String | No | API base URL, defaults to `https://api.flashcat.cloud`, can also be set via the `FLASHDUTY_BASE_URL` environment variable |

## First example: create a team and channel

Add the following resource definitions to `main.tf` to create a team and a channel under that team:

```hcl
# Create an SRE team
resource "flashduty_team" "sre" {
  team_name   = "SRE Team"
  description = "Site Reliability Engineering team"
}

# Create a channel under the SRE team
resource "flashduty_channel" "production" {
  channel_name = "Production Alerts"
  description  = "Channel for production environment alerts"
  team_id      = tonumber(flashduty_team.sre.id)
}
```

## Deploy

<Steps>
  <Step title="Preview changes">
    Run `terraform plan` to preview the resources to be created:

    ```bash
    terraform plan
    ```

    Terraform will display the list of resources planned for creation without actually performing any operations. After confirming the output meets expectations, proceed to the next step.
  </Step>
  <Step title="Apply changes">
    Run `terraform apply` to create the resources:

    ```bash
    terraform apply
    ```

    Terraform will display the plan again and ask you to enter `yes` to confirm execution. After confirmation, the resources will be created.
  </Step>
  <Step title="View results">
    Run `terraform show` to view the current state of managed resources:

    ```bash
    terraform show
    ```

    You can also log in to the Flashduty console to confirm that the team and channel have been successfully created.
  </Step>
</Steps>

## More examples

### Invite members and configure an escalation rule

```hcl
# Invite a new member
resource "flashduty_member_invite" "john" {
  email       = "john@example.com"
  member_name = "John Doe"
}

# Create an escalation rule for the channel
resource "flashduty_escalate_rule" "default" {
  channel_id  = tonumber(flashduty_channel.production.id)
  rule_name   = "Default Escalation"
  description = "Default escalation policy"
  template_id = "your-template-id"

  layers = [
    {
      escalate_window = 15
      target = {
        person_ids = [flashduty_member_invite.john.member_id]
        by = {
          follow_preference = true
          critical          = ["email", "sms", "voice"]
          warning           = ["email", "sms"]
          info              = ["email"]
        }
      }
    }
  ]
}
```

### Destroy resources

When you no longer need these resources, you can destroy them by running the following command:

```bash
terraform destroy
```

<Warning>This operation will delete all resources managed by Terraform. Review carefully before executing.</Warning>

## State management

<Note>
Terraform tracks the mapping between created resources and configuration files through a state file (`terraform.tfstate`).

- **Do not manually modify** the state file.
- **Do not commit the state file to public repositories**, as it may contain sensitive information.
- For team collaboration, it is recommended to use a [remote backend](https://developer.hashicorp.com/terraform/language/backend) (such as S3, Consul, etc.) to store the state file and avoid conflicts.
</Note>
