Flashduty MCP Server provides 6 toolsets with a total of 18 tools. Each tool corresponds to a Flashduty API operation, and the AI assistant automatically selects the appropriate tool and populates parameters based on your natural language instructions.

<Note>
In read-only mode, all write tools (create, update, acknowledge, close, etc.) are disabled. See [Configuration](/en/developer/mcp-server/configuration) for details.
</Note>

## Incident management (incidents)

The incident lifecycle management toolset contains 8 tools, covering incident querying, creation, updating, acknowledging, closing, and similar incident search.

<Accordion title="query_incidents - Query incidents">
Query incidents by ID, time range, status, severity, or channel. Returns complete information including timeline, alerts, and responders.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `incident_ids` | string | No | Comma-separated incident IDs for direct lookup. Other filters are ignored when provided. |
| `progress` | string | No | Filter by status. Options: `Triggered` (triggered), `Processing` (processing), `Closed` (closed). Multiple values separated by commas. |
| `severity` | string | No | Filter by severity. Options: `Info` (Info), `Warning` (Warning), `Critical` (Critical). |
| `channel_id` | number | No | Filter by channel ID. |
| `start_time` | number | No | Query start time, Unix timestamp (seconds). Required when `incident_ids` is not specified. Maximum span of 31 days. |
| `end_time` | number | No | Query end time, Unix timestamp (seconds). Required when `incident_ids` is not specified. |
| `title` | string | No | Search by incident title keyword. |
| `limit` | number | No | Maximum number of results, range 1-100, default 20. |
| `include_alerts` | boolean | No | Whether to include alert preview (first 20 alerts and total count), default `true`. |

**Usage example**

```
Query all triggered Critical severity incidents in the last 24 hours
```
</Accordion>

<Accordion title="query_incident_timeline - Query incident timeline">
Query incident timeline events, including creation, assignment, acknowledgment, closure, notification, and comment records.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `incident_ids` | string | Yes | Comma-separated incident IDs. Event types include: `i_new` (created), `i_assign` (assigned), `i_ack` (acknowledged), `i_rslv` (closed), `i_notify` (notified), `i_comm` (commented), `i_r_*` (field updated). |

**Usage example**

```
View the complete processing timeline for incident INC-12345
```
</Accordion>

<Accordion title="query_incident_alerts - Query incident alerts">
Query alerts associated with an incident. Returns alert title, severity, status, and labels.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `incident_ids` | string | Yes | Comma-separated incident IDs. |
| `limit` | number | No | Maximum number of alerts per incident, range 1-100, default 20. |

**Usage example**

```
List all associated alerts for incident INC-12345
```
</Accordion>

<Accordion title="create_incident - Create an incident">
Create a new incident with a specified title and severity. Optionally associate it with a channel or assign responders.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `title` | string | Yes | Incident title, 3-200 characters. |
| `severity` | string | Yes | Severity level. Options: `Info` (Info), `Warning` (Warning), `Critical` (Critical). |
| `channel_id` | number | No | Associated channel ID. |
| `description` | string | No | Incident description, maximum 6144 characters. |
| `assigned_to` | string | No | Comma-separated member IDs to assign as responders. Use `query_members` to find member IDs. |

**Usage example**

```
Create a Critical severity incident titled "Database replication lag exceeds 30 seconds" with description "MySQL replication lag is continuously growing"
```
</Accordion>

<Accordion title="update_incident - Update an incident">
Update an incident's title, description, severity, or custom fields. Only the provided fields are updated.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `incident_id` | string | Yes | The incident ID to update. |
| `title` | string | No | New title, 3-200 characters. |
| `description` | string | No | New description, maximum 6144 characters. |
| `severity` | string | No | New severity level. Options: `Info`, `Warning`, `Critical`. |
| `custom_fields` | string | No | Custom field updates in JSON format. e.g., `{"field_name": "value"}`. Field name format: `^[a-z][a-z0-9_]*$`. Use `query_fields` to look up available fields. |

**Usage example**

```
Escalate incident INC-12345 severity from Warning to Critical and update the description to "Impact has expanded to all users"
```
</Accordion>

<Accordion title="ack_incident - Acknowledge an incident">
Acknowledge an incident, changing its status from triggered to processing.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `incident_ids` | string | Yes | Comma-separated incident IDs. The acknowledgment is recorded in the timeline with the operator's identity. |

**Usage example**

```
Acknowledge incidents INC-12345 and INC-12346
```
</Accordion>

<Accordion title="close_incident - Close an incident">
Close (resolve) an incident, changing its status to closed.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `incident_ids` | string | Yes | Comma-separated incident IDs. The closure is recorded in the timeline with the operator's identity. |

**Usage example**

```
Close incident INC-12345
```
</Accordion>

<Accordion title="list_similar_incidents - Find similar incidents">
Find historically similar incidents based on a specified incident, helping with troubleshooting and reviewing past solutions.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `incident_id` | string | Yes | Reference incident ID used to find similar historical incidents. |
| `limit` | number | No | Maximum number of results, range 1-100, default 20. |

**Usage example**

```

```
</Accordion>

## Change management (changes)

The change record query toolset contains 1 tool for querying deployment, configuration, and other change records to assist with incident-change correlation analysis.

<Accordion title="query_changes - Query change records">
Query change records with filtering by ID, channel, time range, and type. Returns enriched data including channel names and creator information.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `change_ids` | string | No | Comma-separated change IDs for direct lookup. |
| `channel_id` | number | No | Filter by channel ID. |
| `start_time` | number | No | Query start time, Unix timestamp (seconds). Maximum span of 31 days. Defaults to 1 hour ago. |
| `end_time` | number | No | Query end time, Unix timestamp (seconds). Defaults to current time. |
| `type` | string | No | Filter by change type. |
| `limit` | number | No | Maximum number of results, range 1-100, default 20. |

**Usage example**

```
Query all change records in the last hour
```
</Accordion>

## Status pages (status_page)

The status page management toolset contains 4 tools for querying status pages, viewing change events, creating incidents, and updating timelines.

<Accordion title="query_status_pages - Query status pages">
Query status page configuration information, including component lists. Supports filtering by ID or listing all status pages.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `page_ids` | string | No | Comma-separated status page IDs. Returns all status pages when not provided. |

**Usage example**

```
List all status pages and their component information
```
</Accordion>

<Accordion title="list_status_changes - Query status page change events">
Query active incidents or maintenance events on a status page. Returns a list of unresolved/incomplete events.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `page_id` | number | Yes | Status page ID. |
| `type` | string | Yes | Change event type. Options: `incident` (incident), `maintenance` (maintenance). |

**Usage example**

```
Query currently active incident events on status page 123
```
</Accordion>

<Accordion title="create_status_incident - Create a status page incident">
Create an incident event on a status page, specifying affected components and status updates.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `page_id` | number | Yes | Status page ID. |
| `title` | string | Yes | Incident title, maximum 255 characters. |
| `message` | string | No | Initial update message describing the incident. |
| `status` | string | No | Initial status. Options: `investigating`, `identified`, `monitoring`, `resolved`. Default `investigating`. |
| `affected_components` | string | No | Affected components in `ComponentID:Status` format, multiple separated by commas. Available statuses: `degraded`, `partial_outage`, `full_outage`. |
| `notify_subscribers` | boolean | No | Whether to notify status page subscribers, default `true`. |

**Usage example**

```
Create an incident on status page 123 titled "API response latency", status "investigating", affected component comp_001 with status degraded
```
</Accordion>

<Accordion title="create_change_timeline - Add a change timeline entry">
Add a timeline update to an incident or maintenance event on a status page. Can also update the status and affected components.

**Parameters**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `page_id` | number | Yes | Status page ID. |
| `change_id` | number | Yes | Change event ID (incident or maintenance). |
| `message` | string | Yes | Update message describing current progress. |
| `at` | number | No | Update time, Unix timestamp (seconds). Defaults to current time. |
| `status` | string | No | New status. Incident options: `investigating`, `identified`, `monitoring`, `resolved`. Maintenance options: `scheduled`, `ongoing`, `completed`. |
| `component_changes` | string | No | Component status changes in JSON array format. e.g., `[{"component_id":"xxx","status":"degraded"}]`. Available statuses: `operational`, `degraded`, `partial_outage`, `full_outage`. |

**Usage example**

```
Update change event 456 on status page 123 to status "identified" and add message "Root cause identified as database connection pool exhaustion"
```
</Accordion>
